# APPLICATION THREAT MODEL

## ASSETS

The following assets require protection within the AI Nutrition-Pro application:

1.  API Keys: Unique API keys used by Meal Planner applications to authenticate with the AI Nutrition-Pro system.
2.  Dietitians' Content Samples: Proprietary or sensitive content provided by dietitians, used for training or fine-tuning AI responses. Stored in the API database.
3.  LLM Interaction Data: Requests sent to ChatGPT and responses received. This may include proprietary prompt engineering and potentially sensitive information if user inputs contain it. Stored in the API database.
4.  Client Configuration Data: Configuration settings for onboarded Meal Planner applications and tenants. Stored in the Control Plane Database.
5.  Billing Data: Information related to client usage and billing. Stored in the Control Plane Database.
6.  System Configuration Data: Configuration parameters for the AI Nutrition-Pro application itself, managed by the Administrator via the Web Control Plane.
7.  Administrator Credentials: Credentials used by the Administrator to access the Web Control Plane.
8.  Intellectual Property: The logic within the API Application, specific algorithms, and the engineered prompts used for interacting with ChatGPT.

## TRUST BOUNDARIES

1.  TB1: Meal Planner Application (External, Untrusted) <-> API Gateway (Internal, AI Nutrition-Pro System Entry Point)
2.  TB2: Administrator's Environment (External, User-controlled) <-> Web Control Plane (Internal, AI Nutrition-Pro Admin Interface)
3.  TB3: API Application (Internal, AI Nutrition-Pro Core Logic) <-> ChatGPT-3.5 (External System, Third-Party Service)
4.  TB4: API Gateway (Internal, Edge Service) <-> API Application (Internal, Backend Service)
5.  TB5: Web Control Plane (Internal, Application Logic) <-> Control Plane Database (Internal, Data Store)
6.  TB6: API Application (Internal, Application Logic) <-> API Database (Internal, Data Store)
7.  TB7: Internet (Untrusted) <-> Administrator's access point to Web Control Plane (Trusted)

## DATA FLOWS

1.  DF1: Meal Planner Application -> API Gateway (Content: AI generation request, dietitian's content samples; Protocol: HTTPS/REST) - Crosses TB1
2.  DF2: API Gateway -> Meal Planner Application (Content: AI generated results; Protocol: HTTPS/REST) - Crosses TB1
3.  DF3: API Gateway -> API Application (Content: Forwarded request from Meal Planner; Protocol: HTTPS/REST) - Crosses TB4
4.  DF4: API Application -> API Gateway (Content: Response for Meal Planner; Protocol: HTTPS/REST) - Crosses TB4
5.  DF5: Administrator -> Web Control Plane (Content: Configuration commands, login credentials; Protocol: Assumed HTTPS) - Crosses TB2, TB7
6.  DF6: Web Control Plane -> Administrator (Content: Configuration UI, billing data; Protocol: Assumed HTTPS) - Crosses TB2, TB7
7.  DF7: API Application -> ChatGPT-3.5 (Content: Prompts, content samples for LLM processing; Protocol: HTTPS/REST) - Crosses TB3
8.  DF8: ChatGPT-3.5 -> API Application (Content: LLM-generated content; Protocol: HTTPS/REST) - Crosses TB3
9.  DF9: Web Control Plane -> Control Plane Database (Content: Client data, config, billing data - read/write; Protocol: TLS) - Crosses TB5
10. DF10: API Application -> API Database (Content: Dietitian's content samples, LLM requests/responses - read/write; Protocol: TLS) - Crosses TB6

## APPLICATION THREAT MODEL

### ASSETS

*   **User Personal Data**: This includes age, weight, dietary preferences, historical health data, and any other Personally Identifiable Information (PII).
*   **Nutrition Data**: This includes recipes, meal plans, and proprietary nutritional information.
*   **User Account Credentials**: Usernames, passwords, and authentication tokens used for accessing the system.
*   **AI Models**: The core algorithms used for generating personalized nutrition recommendations.
*   **API Keys**: Credentials used for communication with external services.
*   **Session Data**: User session information that could potentially be hijacked.

### TRUST BOUNDARIES

1.  **User's Device/Browser <-> AI Nutrition-Pro UI** (User inputting data, receiving advice)
2.  **UI <-> API Layer** (Communication over the internet)
3.  **API Layer <-> Processing Engine** (Internal communication between backend components)
4.  **Processing Engine <-> Database** (Internal communication with the database)
5.  **Processing Engine <-> External Services** (Communication with third-party APIs)
6.  **AWS Environment <-> Internet** (Access to AWS services and external APIs)

### DATA FLOWS

1.  **User Input (DF1)**: User enters personal data (age, weight, dietary preferences, etc.) into the UI. (UI -> API Layer)
2.  **API Request (DF2)**: UI sends data to the API layer via REST API. (UI -> API Layer)
3.  **Input Validation & Forwarding (DF3)**: API layer validates input and forwards it to the processing engine. (API Layer -> Processing Engine)
4.  **Database Interaction (DF4)**: Processing engine queries the database for user data, nutrition information, and meal plans. (Processing Engine <-> Database)
5.  **External Service Interaction (DF5)**: Processing engine interacts with external services for recipe suggestions and nutritional databases. (Processing Engine <-> External Services)
6.  **Recommendation Generation (DF6)**: Processing engine processes data and generates personalized nutrition advice and meal plan. (Internal to Processing Engine)
7.  **Results to API (DF7)**: Results are sent back from the Processing Engine to the API Layer. (Processing Engine -> API Layer)
8.  **Results to UI (DF8)**: API layer sends results to the UI. (API Layer -> UI)
9.  **Progress Tracking (DF9)**: User inputs progress data, which follows a similar path as DF1-DF4. (UI -> API Layer -> Processing Engine -> Database)

## APPLICATION THREAT MODEL

### ASSETS

1.  **User Personal Data**: This includes age, weight, dietary preferences, historical health data, and any other Personally Identifiable Information (PII).
2.  **Nutrition Data**: This includes recipes, meal plans, and proprietary nutritional information.
3.  **User Account Credentials**: Usernames, passwords, and authentication tokens (OAuth 2.0 with OIDC).
4.  **AI Models**: The core algorithms and intellectual property within the Processing Engine used for generating personalized nutrition recommendations.
5.  **API Keys**: Credentials used for communication with external services.
6.  **Session Data**: User session information (e.g., session tokens, cookies) that, if compromised, could lead to unauthorized access.
7.  **Payment Information (Future)**: If the "integration with grocery delivery services" involves payment processing, this will become a critical asset.

### TRUST BOUNDARIES

1.  **User's Device/Browser <-> AI Nutrition-Pro UI**: Interaction point for users with the application.
2.  **UI <-> API Layer (API Gateway)**: Communication over the internet, typically HTTPS.
3.  **API Layer (API Gateway) <-> API Layer (Backend Service)**: Internal communication, potentially within the same network or VPC.
4.  **API Layer (Backend Service) <-> Processing Engine**: Internal communication between backend components.
5.  **Processing Engine <-> Database (PostgreSQL on RDS)**: Internal communication with the database.
6.  **Processing Engine <-> External Services (Third-party APIs)**: Communication over the internet to external data providers.
7.  **AWS Environment <-> Internet**: General boundary for all cloud-hosted components.

### DATA FLOWS

1.  **DF1: User Data Input**: User -> UI (User enters personal data, dietary preferences, health data).
2.  **DF2: API Request (User Data)**: UI -> API Layer (Transmits user data via REST API).
3.  **DF3: Input Validation & Forwarding**: API Layer -> Processing Engine (Validated data is passed to the processing core).
4.  **DF4: Database Interaction (Read/Write)**: Processing Engine <-> Database (Stores and retrieves user profiles, nutrition data, meal plans, progress).
5.  **DF5: External Service Interaction**: Processing Engine -> External Services (Requests recipe suggestions, nutritional information).
6.  **DF6: External Service Response**: External Services -> Processing Engine (Receives data from third-party APIs).
7.  **DF7: Recommendation Generation**: Processing Engine (Internal processing, application of AI algorithms).
8.  **DF8: Results to API Layer**: Processing Engine -> API Layer (Sends personalized nutrition advice and meal plans).
9.  **DF9: Results to UI**: API Layer -> UI (Displays recommendations and plans to the user).
10. **DF10: Progress Tracking**: User -> UI -> API Layer -> Processing Engine -> Database (User updates progress, which is then stored).
11. **DF11: Authentication Flow (OIDC)**: User -> UI -> Authentication Provider (OIDC) -> UI -> API Layer (Token exchange and validation).
